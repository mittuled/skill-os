# Training Design Framework

## Purpose

A framework for designing internal training materials that produce measurable competency gains, not just content consumption.

## Audience-First Classification

Before creating any material, classify the target audience:

| Audience | Primary Need | Knowledge Entry Level | Format Preference |
|----------|--------------|-----------------------|-------------------|
| Support agents | Troubleshooting steps, FAQ responses, escalation paths | Variable — new hires to experienced | Written reference + quick-reference card |
| Sales team | Feature value propositions, demo scripts, objection handling | Product-aware but not deep | Scenario-based guide + demo script |
| Customer success | Workflow walkthroughs, configuration options, common setups | Moderate — power users | Annotated walkthrough + video |
| New hires (all roles) | Foundational product knowledge, core workflows | None — no prior context | Structured module with verification |
| Internal ops / admin | Process procedures, policy compliance | Moderate — role-specific | Step-by-step procedure guide |

---

## Learning Objective Writing Standard

Every training module must have a learning objective written in measurable terms:

**Format**: "After completing this module, [audience] will be able to [action verb] [what] [condition/standard]."

**Action verbs by level** (Bloom's Taxonomy):

| Level | Verbs | Use When |
|-------|-------|---------|
| Remember | Identify, recall, list | Foundational knowledge the learner must hold |
| Understand | Explain, summarise, describe | Conceptual understanding of how something works |
| Apply | Demonstrate, use, execute, complete | Procedural task that must be performed correctly |
| Analyse | Differentiate, compare, diagnose | Troubleshooting and decision-making scenarios |
| Evaluate | Judge, recommend, assess | Experienced audiences making quality calls |

Avoid vague objectives like "understand" or "know about" — replace with a measurable verb.

---

## Module Structure Signal Table

| Learning Objective Type | Recommended Module Structure |
|------------------------|------------------------------|
| Remember — factual | Overview → Key terms → Reference table → Quick quiz |
| Understand — conceptual | Problem/context → Explanation → Example → Check for understanding |
| Apply — procedural | Goal statement → Step-by-step walkthrough → Annotated screenshots → Practice scenario |
| Analyse — diagnostic | Scenario intro → Decision framework → Worked example → Practice case |

---

## Format Selection by Content Type

| Content Type | Best Format | Why |
|-------------|------------|-----|
| Step-by-step procedure | Numbered written guide + annotated screenshots | Learner can reference during execution |
| Conceptual overview | Short written explainer (< 500 words) + diagram | Concepts are absorbed better with visual support |
| Demo or sales scenario | Script with speaker notes + screenshot deck | Sales and CS need exact language, not improvisation |
| Complex workflow | Screen recording + written companion | Video shows interaction nuance; text allows skimming |
| Knowledge verification | Quiz (3–8 questions per module) | Identifies gaps before the learner performs in the field |
| Quick reference | One-page card (PDF or internal wiki page) | Supports recall during live calls or support tickets |

---

## Knowledge Verification Standards

Each module that requires Apply-level competency must include verification:

| Verification Type | Use When | Passing Standard |
|------------------|---------|-----------------|
| Multiple choice quiz | Foundational knowledge check | ≥ 80% correct |
| Scenario-based exercise | Apply-level skills | Must correctly complete all required steps |
| Role-play or demo simulation | Sales / CS use cases | Assessed against scenario scoring rubric |
| Short written response | Diagnostic / analytical skills | Reviewed by manager or trainer |

Do not present verification as a test — frame it as a practice scenario or check-in to reduce anxiety and increase engagement.

## Pilot Feedback Framework

Run a pilot with 3–5 members of the target audience before finalising materials:

| Feedback Dimension | Question to Ask Pilot Group | Decision Threshold |
|-------------------|-----------------------------|-------------------|
| Clarity | "Was anything confusing or unclear?" | Any critical confusion → revise before wider rollout |
| Completeness | "Was anything missing that you needed?" | Any workflow gap → add before wider rollout |
| Relevance | "Was all the content applicable to your role?" | > 20% rated irrelevant → segment or shorten |
| Confidence post-training | "Do you feel ready to [perform the task]?" (1–5 scale) | Mean < 3.5 → revision required |
| Format preference | "Was the format easy to follow?" | < 70% positive → explore alternative format |

## Maintenance Cadence

| Trigger | Action | Owner |
|---------|--------|-------|
| Product release affecting trained workflow | Update affected modules within 5 business days | Content Designer |
| Support spike on a trained task | Review training material for gaps | Content Designer + Support Lead |
| Quarterly content audit | Verify all screenshots match current UI | Content Designer |
| New hire feedback indicates confusion | Prioritise revision of flagged module | Content Designer |
| Feature deprecation | Archive deprecated module; update any referring materials | Content Designer |
