## Technical Discovery Question Bank

[Structured questions organized by evaluation phase.

GUIDANCE:
- Good: Questions grouped by phase (Environment Assessment, Use Case Validation, Decision Criteria Mapping), each tagged with what it uncovers and "listen for" guidance. Minimum 5 questions per phase.
- Bad: "Ask technical questions"
- Format: Table per phase with columns: Question, What It Uncovers, Listen For, Red Flags. Reference: `references/framework.md`]

## POC Qualification and Scoping Framework

[Entry criteria, scope boundaries, and exit criteria for POCs.

GUIDANCE:
- Good: Entry criteria table (minimum ACV, MEDDIC score, executive sponsor requirement), scope template (max 3 use cases, max duration, feature set boundaries), success metric definition method (measurable, pre-agreed, binary pass/fail), exit criteria
- Bad: "Qualify POCs before starting"
- Format: Three subsections (Entry, Scope, Exit) with structured criteria tables]

## Demo Frameworks

[Persona-specific demo structures.

GUIDANCE:
- Good: One framework per persona (Technical Evaluator, Business Buyer, End User) with: demo objective, key messages, feature showcase order, talk track bullets, environment requirements, and common questions to prepare for
- Bad: "Demo the product"
- Format: One subsection per persona with consistent structure. Include setup checklist per demo type.]

## Technical Objection Library

[Common technical objections with evidence-based responses.

GUIDANCE:
- Good: Table with objection, category (security/scalability/integration/deployment), frequency (high/medium), evidence-based response (certifications, benchmarks, reference architectures, pre-built connectors), and proof point source
- Bad: "Handle technical concerns"
- Format: Table sorted by frequency, with response column citing specific evidence]

## Competitive Technical Matrices

[Technical comparison against top 3 competitors.

GUIDANCE:
- Good: Matrix with evaluation dimension (architecture, performance, integrations, security, deployment), our capability, competitor capability, advantage (ours/theirs/neutral), and trap questions to expose competitor weakness
- Bad: "We are technically superior"
- Format: One matrix per competitor with consistent dimensions. Include "trap questions" section per competitor.]

## SE Engagement Model

[When and how SEs engage in the deal cycle.

GUIDANCE:
- Good: Table mapping deal tier (Tier 1/2/3) to: stage-gate entry point, maximum SE hours, engagement type (full POC, light demo, async technical review), escalation criteria for complex requirements
- Bad: "SEs help with deals"
- Format: Tier-based table with explicit hour allocations and entry criteria]
