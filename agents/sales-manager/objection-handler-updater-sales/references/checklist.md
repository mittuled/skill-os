# Checklist: Objection Handler Update

## Purpose

Ensures objection handler updates are thorough, field-validated, and properly distributed. Use at each step of the workflow to verify completeness.

## Step 1: Objection Mining Checklist

- [ ] Call recordings reviewed (minimum 10 recordings from last 30 days)
- [ ] CRM closed-lost reasons exported and categorized
- [ ] Slack #sales channel searched for objection mentions (last 30 days)
- [ ] Rep 1:1 notes reviewed for reported objection themes
- [ ] Each objection categorized by type:
  - [ ] Pricing (cost, ROI, budget)
  - [ ] Competition (specific competitor, alternative approach)
  - [ ] Product capability (feature gap, limitation)
  - [ ] Timing (not now, other priorities)
  - [ ] Trust (company size, stability, references)
- [ ] Frequency count recorded per objection (minimum 3 occurrences to include)
- [ ] Deal impact scored per objection (number of deals lost or stalled)

## Step 2: Response Drafting Checklist

Each response must follow the **Acknowledge-Reframe-Evidence** pattern:

- [ ] **Acknowledge**: Response starts by validating the buyer's concern (not dismissing it)
- [ ] **Reframe**: Response reframes the objection in terms of the buyer's desired outcome
- [ ] **Evidence**: Response includes at least one proof point:
  - [ ] Customer case study or metric
  - [ ] Third-party validation (analyst, review site)
  - [ ] Product capability demonstration
  - [ ] Competitive comparison data
- [ ] Response is conversational, not scripted (framework, not a script)
- [ ] Response includes 1-2 follow-up questions to deepen the conversation
- [ ] Response avoids technical jargon unless the buyer is technical

### Response Quality Checks

| Check | Pass Criteria |
|-------|--------------|
| Length | 3-5 sentences per response (not a paragraph, not a one-liner) |
| Tone | Consultative, not defensive or dismissive |
| Specificity | References specific outcomes, not generic claims |
| Adaptability | Works across deal sizes and buyer personas with minor adaptation |
| Proof point freshness | Evidence is from the last 6 months |

## Step 3: Validation and Publish Checklist

- [ ] Responses reviewed by at least 2 top-performing reps
- [ ] Rep feedback incorporated (wording feels natural, proof points are credible)
- [ ] Responses tested against recent real objections (rep role-play or deal review)
- [ ] Changelog written documenting:
  - [ ] New objections added (with trigger context)
  - [ ] Existing responses updated (with reason for change)
  - [ ] Deprecated objections removed (with justification)
- [ ] Updated handler published to playbook repository
- [ ] Team notification sent with summary of changes and 3 key highlights
- [ ] Next update cycle scheduled (recommended: monthly or after major product/competitive changes)
